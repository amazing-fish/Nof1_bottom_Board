# Codex 项目锚点约定

## 核心理念
- 本仓库采用“锚点”思维：通过维护 `ANCHOR.md` 来固定项目上下文、关键约定和最近的决策。
- 在开始任何修改、评审或测试前，请通读 `ANCHOR.md`，确认现有约定和待办事项。
- 在完成改动并通过自测后，及时更新 `ANCHOR.md`，确保锚点反映最新状态。

## 沟通与回复
- 所有面向用户的文字交流使用简体中文。
- 记录命令或脚本时，优先给出可复制执行的完整片段；避免零散的逐条指令。

## 开发流程
1. **对齐目标**：阅读 `ANCHOR.md` 的“当前目标”“约束”两节，确认本次任务范围。
2. **方案设计**：必要时在提交代码前将设计要点写入 `ANCHOR.md` 的“设计笔记”小节。
3. **实现与测试**：编码时遵循文件内现有风格；测试步骤和结果写入提交说明，并追加到 `ANCHOR.md` 的“验证记录”。
4. **影响分析**：改动完成后，在 `ANCHOR.md` 的“潜在影响”中列出可能受影响的模块或风险项。

## 审计与可追溯
- 请在 `docs/audit.log` 中以时间倒序记录每次改动的摘要、涉及文件、测试命令和结论。
- 如果新增自动化脚本或命令，附上用途与执行示例。

## 代码与文件约定
- 代码变更需配套测试或明确说明测试局限。
- 任何生成文件（如构建产物）不得提交。
- 若引入新约定（编码规范、依赖管理等），同步写入 `ANCHOR.md` 并在 `AGENTS.md` 追加或更新对应条款。

## 提交与复查
- 提交信息需包含本次目标、主要改动、测试摘要。
- 在发起 PR 前，对照 `ANCHOR.md` 自查是否所有约定均已满足，必要时补充说明。
